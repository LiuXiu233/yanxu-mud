# 权限模型

## 边界

权限属于应用层，不由网络连接、显示名称或内容文本决定。账户和角色实体通过 `言域:组件/权限` 保存角色组与显式授权；命令执行器只读取已经由权限服务投影到上下文的稳定权限编号。接入层负责把已认证会话绑定到操作者编号，不能接受客户端自行声明的权限列。

权限组件 v1 数据：

```yanju
据【
  「角色」：列【「玩家」】，
  「授权」：列【「青石镇:权限/传送」】
】
```

角色组是配置中的聚合名称，权限必须是 `命名空间:类别/标识` 形式的稳定编号。未知角色组会被拒绝，不能静默降权或提升权限。

## 内置定义

`默认角色定义（）` 提供：

| 角色组 | 有效权限 |
| --- | --- |
| 玩家 | `言域:权限/游戏` |
| 管理员 | 游戏、管理、账户管理、角色管理、权限管理、移动管理、物品管理、装备管理、属性管理 |

稳定权限编号为：

- `言域:权限/游戏`
- `言域:权限/管理`
- `言域:权限/账户管理`
- `言域:权限/角色管理`
- `言域:权限/权限管理`
- `言域:权限/移动管理`
- `言域:权限/物品管理`
- `言域:权限/装备管理`
- `言域:权限/属性管理`

游戏或代码插件可以向 `权限服务` 构造器传入额外角色定义。扩展会覆盖同名映射但不能删除内置角色；每项权限仍须通过稳定编号检查。

## 有效权限

`有效权限（实体，角色定义）` 按以下确定性流程计算：

1. 规范并去重显式授权。
2. 按组件中的角色组查找映射；未知角色组立即失败。
3. 合并所有角色组权限与显式授权。
4. 去重并排序后返回隔离副本。

`构建命令上下文` 在调用方给出的普通值基础上写入主体编号、账户编号和有效权限列。它不会接受或保留客户端伪造的权限。`权限服务.命令上下文` 从仓储重新读取主体，适合命令执行前的权威投影。

## 管理员引导

`引导管理员` 只用于全新世界的首个管理员：

- 目标账户必须以自己的已认证身份发起。
- 目标实体必须含自归属的账户组件；角色实体即使操作者编号相同也不能完成引导。
- `言域.权限.管理员已引导` 世界变量必须不存在。
- 仓储中不能已经存在管理员角色组。
- 权限组件更新、引导标记、事实事件和结构化消息在同一事务提交。

如果存在管理员但引导标记缺失，服务拒绝继续，维护者必须先从快照、事件日志和审计记录确认状态，不能以再次引导覆盖异常。首次引导之后，管理员轮换使用角色组授予和撤销。

## 授权变更

权限服务提供 `授予权限`、`撤销权限`、`授予角色组` 和 `撤销角色组`。操作者实体必须在同一仓储事务中重新读取，并具有 `言域:权限/权限管理`。服务拒绝重复授予、不存在的撤销、未知角色组、缺少权限组件和无权操作。

撤销管理员角色组前，事务会查询当前管理员数量。至少存在第二个管理员时才允许撤销，因此管理员可以安全轮换，但不能移除最后一个管理员。此保护不依赖引导标记，也不能由普通管理参数绕过。

所有成功变更产生事实事件：

- `言域:事件/管理员引导`
- `言域:事件/权限变更`
- `言域:事件/角色组变更`

事件来源是操作者稳定编号，目标是被修改实体；载荷只包含动作、字段和稳定项目编号，不含密码、令牌或连接信息。

## 系统注册

`建立系统（服务）` 返回编号为 `言域:系统/权限`、优先级为 50 的系统实例，可加入通用系统注册表。支持动作：查询权限、命令上下文、引导管理员、授予/撤销权限和授予/撤销角色组。系统仍复用同一权限服务和事务边界，不维护第二份授权状态。

## 测试要求

权限变更必须使用内存、文件和 SQLite 仓储共享的世界事务语义。规格至少覆盖角色组与显式授权合并、未知角色、首次引导、重复引导、无权操作、授权撤销、管理员轮换、末位管理员保护、命令上下文和系统注册表。测试不通过时不得降级为信任调用方提供的权限列。
