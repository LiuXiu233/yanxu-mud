# 实体与组件

## 稳定协议

实体由 `稳定编号 + 原型编号 + 组件集合 + 标签 + 修订号` 组成。编号统一采用 `内容包名:类别/标识`，例如 `基础世界:房间/长安南门`。实体公共实现位于 `src/实体/实体.yx`，组件注册表位于 `src/组件/组件注册表.yx`。

组件在实体快照中使用信封：

```yanju
据【
  「版本」：1，
  「数据」：据【「名称」：「精铁长剑」】
】
```

注册表认识的组件会应用默认值、言据 Schema 和逐版本迁移；不认识的组件信封和额外字段会原样往返，保证内容插件的前向兼容。任何组件值都必须由空、逻辑、数、文字、列和文字键据组成，不能把套接字、任务、回调或其他宿主资源塞入状态。实体标签不能为空或带首尾空白，与仓储标签查询使用同一名称规则。

## 修订与变更集

添加/替换/删除组件、修改原型和标签都会把修订号增加一。变更集记录基础修订、结果修订和有序操作；仓储保存时比较基础修订，冲突返回 `YANYU_REPOSITORY_REVISION_CONFLICT`。实体快照与变更集的完整信封（包括扩展字段）都必须是无环且处于复杂度预算内的普通值。应用前会完整验证每项动作的类别、名称、必需字段及字段类型，畸形动作统一返回 `YANYU_ENTITY_CHANGESET_INVALID`；之后才复制实体并顺序应用，因此任何失败都不会修改原对象。每次本地变更也先物化完整候选变更集并验证候选完整快照，再交换状态和递增修订。变更集接近复杂度上限时应先调用 `确认变更()` 建立新基线。

调用 `确认变更()` 后，当前修订成为下一变更集的基础，已确认操作从内存中移除。快照仍包含完整当前状态，与事件日志的事实记录用途不同。

## 原型

原型也是可序列化实体快照。建立动态实体时，动态组件按稳定编号替换原型组件，标签取并集；之后实例状态独立变化。多层原型解析和循环检查由内容运行时负责，实体协议只处理已经解析的一层快照。

## 首批通用组件

`src/组件/通用组件.yx` 注册 24 个题材无关组件：描述、位置、容器、出口、可移动、物品、属性、状态、背包、装备、战斗、技能、冷却、交互、对话、任务、掉落、AI、账户、角色、权限、频道、定时和持久化。它们都拥有稳定编号、v1 Schema、默认值、序列化边界、迁移入口、中文说明和规格测试。

属性组件的 `基础` 与 `当前` 都是名称到有限数的典，可选 `来源` 列只保存稳定修正源编号。装备组件的 `槽位` 是槽位名称到物品实例编号的典；已装备物品不同时出现在背包编号列中，但其位置仍指向持有角色。这些双向与派生一致性由装备、属性服务在世界事务中维护，不由组件 Schema 单独推断。

状态组件的 `状态` 列保存 `{状态编号，定义编号，来源编号，层数，开始时间，到期时间}` 普通典。静态名称、描述、属性和叠加策略仍来自内容图；`到期时间` 为零表示永久。状态与装备通过同一个属性来源聚合器解析，属性组件不保存两类静态定义副本。

技能组件的 `已学` 列只保存稳定技能定义编号；冷却组件的 `到期` 典把行为或技能编号映射到可注入世界毫秒。战斗组件保存生命、最大生命、攻击、防御、阵营与战斗状态，并可在死亡后保留死亡时间和击败者编号；显示技能数据、伤害参数和技能行为仍来自内容图与代码注册表。死亡标签与零生命由战斗和死亡服务在同一世界事务中维护。

任务组件的 `进度` 典按任务定义编号保存状态、逐步计数、接受时间和完成时间，不复制步骤目标、要求或奖励。对话组件的 `节点` 和 `进度` 保存单个活动会话的定义编号、对象编号、节点索引与开始时间，空节点必须对应空进度。任务与对话服务每次读取都会用当前内容定义校验这些动态引用。

掉落组件的 `表` 和 `次数` 引用静态内容定义，`已结算`、`结算时间` 与 `结算事务` 构成幂等动态状态；旧快照缺少后三项时按未结算规范化，不会因只读查询触发迁移。成功结算与生成的物品实例在同一世界事务提交。

AI 组件的 `行为` 引用代码注册表中的稳定编号，`间隔毫秒` 和 `任务编号` 关联唯一持久计划任务，`参数` 保存行为配置，`状态` 保存可恢复的逐轮结果。函数、仓储对象和时钟不会进入组件；启用、停用与每次行为状态写回都由 AI 服务维护组件和任务的一致性。

核心组件不能写死修仙境界、门派、职业或具体世界观。游戏需要的额外字段可以由内容 Schema 扩展，题材专属组件应使用游戏自己的命名空间。
