# 快照与恢复

世界快照用于灾难恢复、版本升级前保护和确定性测试，不替代持续备份或事件审计。快照只包含动态状态：实体实例、世界变量、计划任务和数据迁移历史。房间、物品原型等静态内容仍由已锁定版本的内容包提供。

## 快照格式

`言域/世界快照` 格式 2 包含：

| 字段 | 含义 |
| --- | --- |
| `编号` | 仓储内唯一的稳定名称 |
| `序号` | 从 1 开始单调递增的快照序号 |
| `元数据` | 原因、操作者、内容包版本等普通值 |
| `状态` | 完整动态状态 |
| `校验和` | 对其余字段确定性 JSON 计算的 SHA-256 |

创建快照时应在元数据中记录现实时间、世界时间、内容图摘要、引擎版本、操作者和工单编号。SHA-256 用于发现损坏，不证明来源，也不替代操作系统访问控制或外部签名。

## 恢复流程

1. 停止新连接输入、命令执行和到期任务领取。
2. 等待当前世界事务提交或回滚，确认没有悬挂事务。
3. 创建恢复前快照，并把当前事件日志尾序号和内容图摘要写入元数据。
4. 调用 `检查快照恢复`，核对快照编号、摘要、对象统计和 `会改变状态`。
5. 确认目标快照对应的内容包版本仍可装载；必要时先准备内容与数据迁移。
6. 调用 `恢复快照`，核对 `已验证` 为真且恢复后状态摘要等于目标摘要。
7. 从仓储重新读取关键实体、世界变量和计划任务；文件或 SQLite 后端还应重开连接复核。
8. 记录管理员事实事件，再恢复世界输入。

恢复以新世代写入，不覆盖或截断历史世代。快照目录和事件日志也保持不变，因此可以审计“恢复到旧状态”这一操作。文件仓储仍受单写者和缺少 `fsync` 的边界约束；生产多写者服务使用 SQLite。

## 公共接口

```yanxu
定 快照：典 为 仓储.创建快照（「发布前」，{「版本」：「0.4.0」}）；
定 检查：典 为 仓储.检查快照恢复（「发布前」）；

若 检查【「会改变状态」】 则
    定 结果：典 为 仓储.恢复快照（「发布前」）；
终
```

`检查快照恢复` 无副作用。若快照不存在、摘要不一致、内部实体或任务无效，预检会在任何状态写入前失败。恢复期间发生并发提交时返回仓储事务冲突；调用方必须重新预检，不可盲目覆盖。

## 备份边界

- 内存仓储快照随进程消失，只用于测试和临时世界。
- 文件仓储应备份整个世代目录，不能只复制 `当前.yj`。
- SQLite 应使用 SQLite 在线备份或停服后一致复制，不能只复制活动 WAL 中的一部分。
- 内容包、引擎制品和配置必须与状态备份使用同一发布清单和校验和归档。
