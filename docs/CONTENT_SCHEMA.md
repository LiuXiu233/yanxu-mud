# 内容 Schema

内容 Schema 在内容进入世界前验证普通数据。它描述静态内容的形状，不执行转换脚本或任意代码；Schema 版本与引擎版本、内容包版本彼此独立。

## 定义外形

协议名为 `言域/内容Schema`，格式版本为 `1`。定义必须恰好包含：

```yanju
据【
  「协议」：「言域/内容Schema」，
  「格式版本」：1，
  「编号」：「我的包:Schema/房间」，
  「版本」：1，
  「适用类别」：「房间」，
  「规则」：据【
    「类型」：「据」，
    「必需」：列【「名称」，「描述」】，
    「允许额外」：假，
    「属性」：据【
      「名称」：据【「类型」：「文」，「最短」：1，「最长」：64】，
      「描述」：据【「类型」：「文」，「最长」：4096】
    】
  】，
  「说明」：「房间静态定义」，
  「扩展」：据【】
】
```

`编号` 必须是稳定编号，`版本` 是正整数。`适用类别` 必须与内容文档类别一致；使用 `*` 可建立共享 Schema。内容文档同时记录 `Schema编号` 和 `Schema版本`，版本不符会在注册时失败。

## 规则语言

每条规则必须显式声明 `类型`。类型可以是单个文字，也可以是无重复的联合类型列：

| 类型 | 含义 |
| --- | --- |
| `任意` | 任意受预算约束的普通值 |
| `据` | 键值对象 |
| `列` | 列表 |
| `文` | 文字 |
| `数` | 有限数值 |
| `理` | 逻辑值 |
| `空` | 空值 |

可用规则字段：

| 字段 | 适用范围 | 语义 |
| --- | --- | --- |
| `类型` | 全部 | 必需；单类型或联合类型 |
| `枚举` | 全部 | 值必须等于枚举中的一项 |
| `最小`、`最大` | 数 | 包含边界的有限数范围 |
| `整数` | 数 | 为真时要求整数 |
| `最短`、`最长` | 文、列、据 | 长度包含边界 |
| `项目` | 列 | 每个项目递归应用的规则 |
| `必需` | 据 | 必须出现的属性名列 |
| `属性` | 据 | 属性名到子规则的映射 |
| `允许额外` | 据 | 是否允许未在 `属性` 声明的键 |

规则最多嵌套 32 层，一个对象规则最多声明 256 个属性和 256 个必需字段。`必需` 中的每个字段必须同时出现在 `属性`。未知规则字段会被拒绝，以避免拼写错误被静默忽略。

## 校验顺序

内容图按以下顺序建立：

1. 规范内容包清单并解析依赖顺序；
2. 规范和注册 Schema；
3. 规范内容文档并检查包归属；
4. 解析跨包引用与原型链；
5. 合并原型数据；
6. 使用内容文档声明的 Schema 对最终合并数据校验；
7. 检查行为编号与本地化键；
8. 生成确定性指纹和内容图摘要。

因此，子原型不能借覆盖绕过 Schema，父原型中的默认字段也会参与最终验证。错误问题最多保留 256 项，只暴露稳定代码、数据路径、消息和预期，不泄露运行时踪迹。

## 注册表 API

```yanxu
引「包:言域/内容Schema」为 内容Schema；

定 表 为 内容Schema.注册表（）；
表.注册（定义）；
表.断验（「我的包:Schema/房间」，房间数据）；
```

`注册表` 支持 `注册`、`替换`、`查询`、`版本`、`适用类别`、`编号列`、`检查`、`验证`、`断验` 和 `断验普通数据`。查询和输出返回隔离副本，调用方不能通过修改返回值篡改注册表。

稳定错误包括：

- `YANYU_CONTENT_SCHEMA_INVALID`：Schema 定义或规则非法；
- `YANYU_CONTENT_SCHEMA_EXISTS`：编号重复；
- `YANYU_CONTENT_SCHEMA_MISSING`：编号不存在；
- `YANYU_CONTENT_SCHEMA_DATA_INVALID`：数据不满足规则；
- `YANYU_CONTENT_DOCUMENT_SCHEMA_MISMATCH`：文档类别或版本与 Schema 不一致。

## 演进规则

0.x 中规则协议仍是预览面。进入 1.0 后，同一 Schema 版本不得改变既有字段含义；新增可选字段可以保持版本，收紧约束、改变必需字段或删除字段必须提高 Schema 版本并提供内容迁移说明。运行中的 Schema 变化不会直接热重载，需要安全重启或迁移。
