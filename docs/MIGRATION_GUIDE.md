# 数据迁移指南

数据迁移只演进动态状态。静态房间、物品原型、任务模板和本地化文本属于内容包，按内容包版本与内容迁移处理；不要把静态内容复制进数据库。

## 定义迁移

每项迁移具有稳定命名空间编号、语义版本、不可随意改写的说明、应用器和后置检查。定义摘要由编号、版本与说明确定。迁移一旦在任何公开环境应用，不得重排、删除或修改；修正逻辑必须追加新编号。

```yanxu
引「包:言域/数据迁移」为 迁移；

法 应用（事务，上下文：典）：典 则
    事务.设置世界变量（「世界.数据版本」，「2」）；
    归 {「已更新」：真}；
终

法 检查（事务，上下文：典）：理 则
    归 （事务.读取世界变量（「世界.数据版本」） 等于 「2」）；
终

定 定义 为 迁移.新迁移（「基础世界:迁移/世界数据二版」，「2.0.0」，「登记世界数据二版」，应用，检查）；
```

应用器和检查器都接收同一个仓储事务与上下文。应用器必须返回普通详情典，记录中会保存该结果、说明和定义摘要。检查器只返回理值；返回假或抛错会回滚整个批次。言序函数体不可确定性序列化，因此定义摘要覆盖编号、版本与说明，不是源码签名；已经发布的应用器或检查器逻辑若需改变，必须追加新迁移编号。

## 启动顺序

1. 装载并验证与目标版本匹配的代码和内容包。
2. 打开仓储，完成世代链、载荷和快照校验。
3. 调用 `计划摘要` 展示待办迁移，并核对备份与维护窗口。
4. 创建升级前世界快照。
5. 调用 `执行`，传入完整、有序的迁移定义列和普通值上下文。
6. 重开持久化后端并再次调用 `计划`，结果必须为空。
7. 运行关键领域查询和升级规格，再开放世界输入。

一个 `执行` 调用中的所有待办迁移只提交一个仓储世代。执行期间其他写入者推进世代会返回 `YANYU_REPOSITORY_TRANSACTION_CONFLICT`；重新读取、重新规划后可以重试。

## 历史分歧

以下情况返回 `YANYU_DATA_MIGRATION_HISTORY_DIVERGED` 并阻止继续：

- 仓储中的已应用记录多于当前定义；
- 历史不是当前定义的严格前缀；
- 序号不从 1 连续递增；
- 记录缺少定义摘要或摘要格式无效；
- 已应用迁移的编号、版本或定义摘要发生变化。

不要通过直接编辑数据库“修好”历史。先保全原数据库与内容制品，确认部署了正确代码；若确需修复，编写独立、可审计的恢复工具和新的前向迁移，并记录管理员事实事件。

## 失败与回滚

应用器、后置检查或历史记录任一失败都会回滚整个批次。应用器不得执行无法由仓储事务回滚的网络调用、文件覆盖或外部服务写入。确需跨系统协调时，迁移只写可靠发件箱，由提交后消费者使用稳定操作编号幂等处理。

在升级失败后，保持世界输入关闭，保存错误代码与追踪，确认仓储世代未推进，然后修复迁移并以新编号追加定义。已经成功提交的迁移不可自动向后撤销；需要回退版本时使用升级前快照，并遵循 [快照恢复流程](SNAPSHOTS.md)。
