# 架构

## 原则

言域以协议隔离变化频率不同的部分：具体游戏依赖引擎，接入和持久化实现依赖稳定协议，世界内核不反向依赖边界实现。

```text
客户端与工具
    ↓ 结构化输入/输出
接入层（连接、编码、限流、恢复）
    ↓ 命令请求/会话上下文
应用层（账户、角色、权限、频道、管理）
    ↓ 世界事务
世界内核（实体、组件、系统、事件、查询、调度）
    ↓ 变更集/仓储协议
内容运行时与持久化
```

## 关键边界

1. 引擎与游戏：引擎只提供通用组件、系统和协议；青石镇通过独立清单、源码和内容包使用它们。
2. 逻辑与内容：规则由言序模块注册，文本、房间、原型、任务与掉落由言据定义；言据不执行任意代码。
3. 静态与动态：静态原型来自内容包；角色、物品实例、任务进度和计划任务进入状态仓储。
4. 接入与世界：网络输入转换为命令请求；世界层只返回结构化消息，渲染器在边界产生纯文本、ANSI 或 Web 数据。
5. 持久化与领域：世界提交变更集；内存、言据文件和 SQLite 仓储实现同一契约。

## 世界轮次

世界循环由事件唤醒，不高频空转。单轮按网络输入、到期任务、命令、世界事务、领域事件、持久化、结构化输出的顺序处理。每一阶段使用有限非负整数硬预算；所有时钟与随机种子可注入，确保测试和重放确定。

一轮只读取一次世界时间。处理器、时钟或调度器故障会转换为不含原始消息和源码位置的结构化阶段错误；唤醒计算失败时采用“仍有工作”的保守结果，已发送的结构化输出不会从轮次报告中丢失。运行锁在全部成功与失败路径上释放，下一轮可以继续恢复处理。

## 稳定编号

内容编号统一为 `内容包名:类别/标识`。协议编号、错误代码和行为编号是存档与插件边界的一部分，不能以显示文字代替。显示文字和本地化键可以演进，稳定编号不得在兼容版本中改义。
