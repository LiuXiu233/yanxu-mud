# 系统与调度

系统注册表位于 `src/系统/系统注册表.yx`。系统以稳定编号注册，把规则表达为接收上下文的处理器；实体只保存状态，不持有长期运行任务。

## 生命周期

一个系统具有编号、优先级、处理器、可选启动器和可选停止器。注册表支持添加、替换、禁用、启用、装饰、排序、查询以及批量启动、处理和停止。

```yanxu
定 移动 为 系统.系统（
    「言域:系统/移动」，
    100，
    处理移动，
    初始化移动，
    停止移动
）；
注册表.注册（移动）；
```

优先级必须是有限整数；数值较小者先处理，相同优先级按稳定编号排序。启动按正序执行，停止按逆序执行，便于后启动的依赖先释放。装饰器接收旧处理器并返回新处理器，可用于追踪、权限或指标，而无需修改规则实现。

系统必须启动后才能处理；停止是终态，防止已经释放资源的实例被误复用。需要重启时应建立并替换一个新实例。

`启动全部` 具有失败清理语义：任一启动器失败时，注册表先清理失败候选，再按逆序停止本次已经启动的实例，并保持整体未运行；尚未轮到的实例不会被启动。由于已清理实例进入“已停止”终态，调用方应修正故障后用新实例建立新的注册表，而不能把半次启动当作可继续运行的世界。清理器异常会作为脱敏后的结构化详情随生命周期错误返回，并且不会阻止其余实例继续清理。

处理器必须是函数，启动器和停止器必须为空或函数；构造时立即验证这些边界。装饰器自身及其产物也都必须是函数，装饰失败不会改写原处理器。这些配置、启动、停止和装饰错误统一使用 `YANYU_SYSTEM_LIFECYCLE`，对外错误详情不会携带回调原始异常消息或源码位置。

## 运行中注册与替换

注册表在 `启动全部` 成功后保存本轮生命周期上下文。此后注册新编号时，新实例会先用该上下文启动，启动成功后才加入注册表；启动失败时映射保持不变，并调用候选停止器清理可能已分配的资源。

`替换` 只接收状态为“已创建”的候选实例，并继承旧实例的启用状态。若注册表正在运行，替换按以下提交顺序执行：

1. 启动候选实例；若失败，清理候选并保留仍在运行的旧实例。
2. 停止旧实例；若失败，停止候选作为回滚，并继续保留旧映射。
3. 两步均成功后才把编号映射切换为候选实例。

因此成功替换不会遗留旧实例资源，失败替换也不会把注册表切换到未启动或已停止的实例。候选启动与旧实例停止之间存在很短的重叠窗口，生命周期回调必须允许同一系统编号的新旧版本短暂并存。若停止器自身失败，注册表会保持原映射和原状态以便重试；回调若已经执行了部分外部副作用，应自行提供幂等清理语义。

对未注册编号调用 `替换` 等同于 `注册`。注册表整体未运行时，替换只切换“已创建”候选，不触发生命周期回调；下一次 `启动全部` 再启动它。

不要绕过注册表直接启动已注册实例。若旧实例已在注册表生命周期之外运行，注册表没有可靠的停止上下文，`替换` 会拒绝切换，调用方必须先用原上下文显式停止旧实例；这条约束防止无上下文替换遗留旧资源。

## 移动与观察

`包:言域/空间` 保持静态房间与动态位置分离。房间解析器接收稳定房间编号，返回内容图的已解析文档或等价测试替身；空间服务只消费名称、描述和出口，不把静态房间复制到仓储。`内容图房间解析器` 把已经验证的内容图适配到同一调用边界。

移动先规范中英文方向别名，再在世界事务中读取实体位置、当前房间和目标房间。不可移动实体、战斗中实体、无出口方向、缺失目标房间和无权跨实体控制都会在修改前失败并回滚。成功提交位置变更、`言域:事件/实体移动` 和目标房间场景消息；事件载荷保留实体、操作者、方向、起点与终点稳定编号。

观察不修改状态。它查询含位置组件的动态实体，排除观察者自身，并按 `物品` 标签选择物品引用，否则使用人物引用。输出是结构化场景描述，正文含房间文本与可见实体列表，出口使用房间引用节点；ANSI、纯文本或 Web 表示仍由消息渲染边界生成。

`建立移动系统` 返回优先级 100 的 `言域:系统/移动`，`建立观察系统` 返回优先级 110 的 `言域:系统/观察`。二者加入同一注册表时，一次共享上下文处理先完成移动，再观察新房间。NPC 或管理工具跨实体移动必须在权威上下文中具有 `言域:权限/移动管理`，不能信任客户端自行附带的权限列。

## 物品、背包与容器

`包:言域/物品` 通过注入的定义解析器读取内容图或测试替身。静态定义包含名称、描述、类型、是否可堆叠、属性、价值和重量；仓储中的实例只保存稳定定义编号、数量、绑定状态、创建时间、位置和其他动态组件。内容热重载因此可以更新非破坏性静态表现，而实例身份和状态保持稳定。

创建物品需要权威 `言域:权限/物品管理`，并在一个世界事务中写入实例及可选动态容器引用。拾取、放下和容器转移同时更新来源列表、目标列表和物品位置；任一容量、权限、来源匹配、修订或提交检查失败都会回滚整次操作。没有动态实体的稳定房间编号可作为地面位置；具有容器组件的动态房间则参与双向一致性检查。容器查看是只读操作，会拒绝悬空实例和位置错配，并返回结构化物品列表与消息。

`言域:系统/物品`、`言域:系统/背包` 和 `言域:系统/容器` 的优先级分别为 120、130 和 140。物品系统负责定义实例的创建与查询，背包系统负责查看、拾取和放下，容器系统负责查看与容器间转移。系统只接受普通典上下文；接入层必须先用权限服务建立权威权限投影，不能把客户端声明的权限直接传入。

## 装备、属性与状态

物品静态定义可用 `装备槽` 文字列声明兼容槽位。穿戴不改变物品的角色归属位置，而是在同一角色实体内把物品编号从背包列移到装备槽典；卸下做相反变更。槽位占用、不兼容、物品归属错配和背包容量会在提交前拒绝，世界事务保证背包、装备、属性和战斗投影不会只更新一部分。

属性组件区分不可由修正源覆盖的 `基础`、计算缓存 `当前` 和稳定 `来源` 编号列。每个修正源形如 `{来源，属性}`；修正源聚合器按稳定提供器编号收集装备、状态及插件来源，再按来源编号排序。同名数值依次相加，因此提供器注册顺序、输入列顺序、仓储后端和执行器不会改变结果。属性值及每一步结果必须是有限数。重算会把攻击、防御和最大生命同步到战斗组件，并在最大生命降低时截断当前生命。

`言域:系统/装备` 优先级为 150，负责查看、穿戴和卸下；`言域:系统/属性` 优先级为 160，负责查询和受权威权限保护的显式重算。装备服务直接在自己的世界事务工作实体上调用属性领域变更，穿戴完成时不需要再排队第二个非原子系统动作。角色本人可以穿脱自己的装备，跨角色装备与直接属性重算分别需要 `言域:权限/装备管理` 和 `言域:权限/属性管理`。

状态静态定义由内容图解析，动态状态只记录实例编号、定义编号、来源编号、层数、开始时间和到期时间。定义支持刷新、叠加和拒绝重复三种策略，层数受稳定上限约束；持续时间为零表示永久，否则以可注入世界毫秒判定。施加、移除和到期清理会在同一事务重算共享聚合器中的全部来源，因此穿脱装备不会删除状态修正，状态变化也不会删除装备修正。`言域:系统/状态` 优先级为 170，写操作需要 `言域:权限/状态管理`。

## 冷却、技能、战斗与死亡

冷却组件保存行为编号到世界毫秒到期值的确定性映射。读取不会改写状态；应用、强制设置和到期清理拒绝负数、非整数、溢出时间和无效编号。`冷却服务` 的管理写操作需要 `言域:权限/冷却管理`，并在世界事务中提交组件、`言域:事件/冷却变更` 和结构化消息。`言域:系统/冷却` 优先级为 190。

技能静态定义由内容图解析，包含名称、描述、行为编号、冷却毫秒、是否需要目标和普通参数；动态技能组件只保存已学定义编号，冷却组件保存使用状态。行为注册表支持注册、替换、移除与确定性编号查询，内容数据不能携带回调。学习和遗忘需要 `言域:权限/技能管理`；本人可使用已学技能，代替其他实体使用也需要该权限。技能行为、冷却、技能事实及行为产生的事实事件和消息共享一个世界事务，行为或提交失败会全部回滚。`言域:系统/技能` 优先级为 180。

战斗按 `max(最低伤害, floor(攻击 * 倍率 / 100) + 固定伤害 - 防御)` 计算有限非负整数伤害，并截断到目标当前生命。攻击者和目标必须不同、存活、具有战斗与位置组件且处于同一位置；本人可攻击，代替其他实体攻击需要 `言域:权限/战斗管理`。双方战斗投影、伤害事实、结构化战斗消息和致死时的死亡标签、死亡时间、击败者编号及死亡事实原子提交。`言域:技能行为/伤害` 把同一领域操作接入技能行为注册表；`言域:系统/战斗` 优先级为 200。

死亡服务把生命为零或具有死亡标签的实体视为死亡。本人可以复活自己，跨实体复活需要 `言域:权限/死亡管理`；恢复生命为零时恢复至最大生命，正值截断到最大生命，可选复活房间只在实体具有位置组件时应用。生命、战斗状态、死亡元数据、标签、位置、`言域:事件/实体复活` 和结构化消息在同一事务提交，任何后置检查失败都不留下部分复活状态。`言域:系统/死亡` 优先级为 210。

## 任务与对话

任务静态定义包含名称、描述、一个或多个 `{类型，目标，数量}` 步骤和普通奖励描述；任务组件只保存规范化逐步计数与接受/完成时间。本人可以接受和放弃进行中的任务，跨实体操作需要 `言域:权限/任务管理`；推进始终需要该权限，调用方必须使用权限服务产生的可信投影，不能让客户端自行声明击败或收集事实。推进量截断到步骤剩余要求，全部步骤完成时同事务写入状态并发布 `言域:事件/任务完成`。事件中的奖励只是描述，物品或货币授予必须由组合规则明确执行，不能把“已完成”误报成“已发奖”。`言域:系统/任务` 优先级为 220。

对话静态定义保存唯一有序节点与文字，动态组件最多保存一个活动会话。本人可以开始、推进或结束自己的对话，跨实体控制需要 `言域:权限/对话管理`；死亡实体不能开始新对话。每次转换原子更新节点、定义、对象、索引与开始时间，并提交 `言域:事件/对话变更` 和只含结构化人物引用与文字的消息。通用服务不假定 NPC 必须是动态实体，也不自行判断距离；游戏命令在调用开始前负责把内容 NPC、动态实例和空间规则组合起来。`言域:系统/对话` 优先级为 230。

## 掉落

掉落表保留在内容图，每项声明稳定物品定义编号、零到一概率及正整数数量范围。`抽取掉落` 不读取全局随机状态，而是向注入函数传递掉落表、抽取序号、条目序号、物品编号、随机种类和排他上限；返回值必须是上限内的有限非负整数。概率使用固定百万刻度，数量只在范围大于零时追加一次随机请求，因而同一内容、调用顺序和随机序列产生相同结果。

动态掉落组件保存表、次数、是否已结算、结算世界时间和事务编号。结算始终需要 `言域:权限/掉落管理`，并通过注入的实例编号生成器建立稳定物品实例；堆叠定义保留抽中数量，非堆叠定义展开为多个数量一实例。来源状态、全部物品、容器反向引用、`言域:事件/掉落结算` 与结构化消息共享一个世界事务；任何实例重复、容量不足或回调失败都不留下部分结果。安全预算限制每表 128 项、每次结算 100 轮、256 个命中结果和 256 个动态实例。`言域:系统/掉落` 优先级为 240。

## 可注入时间与计划任务

`src/测试支持/虚拟时钟.yx` 提供确定性虚拟时钟和调度器。调度器保存普通值载荷，按到期时间和任务编号排序，支持一次性任务、重复任务、取消、预算、快照与恢复。

时钟起点、推进量、设置值、任务到期时间和调度时的当前时间均须为有限非负数，可以使用小数毫秒；重复间隔和每次取出预算须为有限非负整数。正推进量必须使时钟严格增长，正间隔也必须使下次到期严格晚于本次；因浮点精度在 `2^53` 一类大值处停滞时，调度器会在改动状态前稳定拒绝。任务编号不可为空、全空白或含首尾空白。`NaN` 和无穷大在入口处被拒绝，不会进入排序、预算比较或持久状态。

调度快照以单个任务为复杂度预算与深复制边界，不把整个世界的任务列表误当成一个 10000 节点普通值。恢复会先用与新建任务相同的规则预验证全部项目、重复编号和排序，再一次交换调度状态；任何失败都保留原任务表。`到期` 也在独立工作列表上物化本批输出和重复任务，本批全部成功后才交换状态；后一项失败不会吞掉已预处理的前一项。输入快照和返回快照也都与调度器内部状态隔离。

生产世界把现实或单调时钟适配到同一“当前毫秒”调用边界；测试直接推进虚拟时钟。冷却、刷新、任务重置、NPC 行为、战斗回合、世界时间和公告都应登记计划任务，不为每个实体创建常驻任务，也不以 30/60 帧空转轮询。

## 系统编写约束

- 通过上下文访问仓储、事务、事件和结构化消息，不直接访问套接字或数据库实现。
- 所有可持久状态写入实体组件、世界变量或计划任务，不藏在不可序列化的闭包中。
- 随机规则接收显式随机源或种子，确保测试和重放可复现。
- 一次处理遵守世界轮次预算；长工作拆成可恢复的计划任务。
