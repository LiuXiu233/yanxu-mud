# 领域事件

事件总线位于 `src/事件/事件总线.yx`。它只处理普通言序值和回调，不依赖网络、数据库或任何渲染格式。

## 事件信封

`新事件` 建立一个稳定信封，字段包括事件编号、全局序号、世界时间、现实时间、类别、事件种类、来源、目标、载荷、命令编号、事务编号、追踪编号和内容包版本。类别只能是 `意图`、`事实` 或 `系统`；事件种类使用 `内容包名:类别/标识` 格式。

新事件的全局序号为零。`发布` 和 `入队` 只接受这种未编号事件；总线在事件首次进入时分配严格递增的正整数序号，因此入队事件不会因延后排空而被后来的同步事件抢占序号。分配前还会验证后继序号在数值表示中严格增长；到达浮点整数精度边界时稳定拒绝，不会复用全局序号。已编号事件必须使用独立的 `重放` 入口。整个事件信封（包含额外字段）都必须是无环、可序列化且处于复杂度预算内的普通值。信封在入队、分发、历史查询、过滤器和处理器边界均深复制，回调不能借可变引用篡改日志或其他订阅者的输入。

## 订阅与顺序

```yanxu
定 总线 为 事件.总线（16，10000）；
定 订阅编号：文 为 总线.订阅（
    「青石镇:事件/进入房间」，
    处理进入，
    100，
    只看玩家，
    假
）；
```

优先级必须是有限整数，数值较小者先执行；相同优先级严格按注册顺序执行。处理器必须是函数，过滤器必须为空或函数。空事件种类表示订阅全部事件。过滤器接收事件副本；它的篡改不会泄漏给当前处理器或后续订阅，过滤失败也只跳过当前订阅并写入踪迹。处理器结果必须是普通值并在返回边界复制；回调失败只返回公开错误类别，不暴露原始异常文字或源码位置。单次订阅在第一次匹配后、调用处理器前即撤销，因此处理器同步递归发布同种事件也不会再次触发它。

## 同步、队列与重放

- `发布` 同步调用匹配的处理器，返回每个处理器的结构化结果。
- `入队` 保存事件副本；`排空` 按先进先出顺序处理指定数量。
- `重放` 接受已编号的历史事件列，先原子验证整列，再保留协议字段并附加重放标记后分发。输入全局序号必须是正整数且严格递增，不得与该总线已接受的序号重复或向后回退；末项还必须存在可精确表示且严格更大的后继序号。成功重放会把总线的下一序号推进到末个历史序号之后。
- `历史` 按全局序号升序返回已经开始分发的事件副本，因此即使排队事件晚于同步事件分发，返回值仍可直接交给新总线重放。历史和踪迹都按项复制，不把聚合列表误当成单个普通值。

每个顶层发布批次有最大事件预算，嵌套发布共享该预算。递归深度、事件预算和排空上限都使用有限整数。超过边界分别使用 `YANYU_EVENT_BUDGET_EXCEEDED` 和 `YANYU_EVENT_RECURSION_LIMIT`，防止插件或系统形成无限事件链。

内存历史默认最多保留最新 10000 条。`设置历史上限` 可在运行前调整；设为零会禁用内存历史，降低长服常驻内存。完整审计日志应由事务提交后的持久化事件日志承担，而不是依赖无限增长的进程内列表。

## 事务边界

命令和系统应先在世界事务中暂存事实事件。只有状态变更成功提交后才把事件交给总线；回滚事务不得泄漏事实事件、消息或持久化副作用。事件日志记录提交后的重要事实，不把每个内部意图都当作完整事件溯源数据。
